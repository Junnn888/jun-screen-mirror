# ScreenBridge — Build & Development

> Phase 0 build instructions. Product spec: [`../plans/PLAN.md`](../plans/PLAN.md)
> §5, §12.

## Prerequisites

| Tool | Version (tested) | Used for |
|---|---|---|
| Rust (stable) | 1.95+ (rust-version floor 1.77) | the `core/` workspace + `cbindgen` (build-dep) |
| .NET SDK | 8.0+ (10.0 also works via roll-forward) | C# interop library + ping tests; WinUI 3 app (Windows) |
| Xcode / Swift | 15+ / Swift 5.9+ (tested Xcode 26, Swift 6.3) | macOS SwiftUI app shell (SPM) |
| Windows + Windows App SDK | windows-latest CI image, or VS 2022 17.10+ | building/running the WinUI 3 app shell |

`cbindgen` does **not** need to be installed separately — it is a build
dependency of the `screenbridge-ffi` crate and runs automatically during
`cargo build`.

## The C ABI header

`include/screenbridge.h` is generated by `core/crates/ffi/build.rs` (via the
cbindgen crate) on every `cargo build`. It is committed to the repo so editors
and the apps have it without a prior build; it is deterministic, so rebuilding
should not produce spurious diffs.

## Build & test the Rust core

```sh
cargo build   --manifest-path core/Cargo.toml          # builds libs + regenerates the header
cargo test    --manifest-path core/Cargo.toml          # unit tests
cargo clippy  --manifest-path core/Cargo.toml --all-targets -- -D warnings
cargo fmt     --manifest-path core/Cargo.toml --all -- --check
```

Outputs in `core/target/<profile>/`:
`libscreenbridge_ffi.a` (static), `libscreenbridge_ffi.dylib`/`.so` /
`screenbridge_ffi.dll` (dynamic).

## macOS app shell (SwiftUI / SPM)

Build the core first (the Swift package links its output and `include/`). To
match the package's macOS-13 floor and avoid a deployment-target warning, build
the native lib with `MACOSX_DEPLOYMENT_TARGET=13.0`:

```sh
MACOSX_DEPLOYMENT_TARGET=13.0 cargo build --manifest-path core/Cargo.toml -p screenbridge-ffi
swift build --package-path apps/macos                  # builds the empty-window app shell
swift test  --package-path apps/macos                  # Swift -> core ping test
```

`Package.swift` finds `include/` and `core/target/<profile>/` by absolute path
relative to the manifest. Set `SCREENBRIDGE_CORE_PROFILE=release` to link the
release build (CI does this).

## C# interop & ping test (any OS)

The `ScreenBridge.Interop` library (net8.0) holds the P/Invoke bindings and is
referenced by both the WinUI app and the tests. The tests resolve the native lib
from the cargo output, so they run on macOS/Linux too:

```sh
cargo build --manifest-path core/Cargo.toml            # produce the native lib
dotnet test apps/windows/ScreenBridge.Interop.Tests/ScreenBridge.Interop.Tests.csproj
```

You can point the resolver explicitly with
`SCREENBRIDGE_CORE_LIB=/abs/path/to/libscreenbridge_ffi.dylib` (CI sets this).
On a box whose newest runtime is greater than .NET 8, the test project's
`RollForward=LatestMajor` lets the net8.0 test host run.

## Windows WinUI 3 app shell (Windows only)

Build the core for the host first so `screenbridge_ffi.dll` is copied beside the
app `.exe`, then build the unpackaged WinUI 3 app. **WinUI cannot build AnyCPU —
pass a concrete platform _and_ an explicit RuntimeIdentifier.** On SDKs newer than
8.0 (which `windows-latest` preinstalls) `-p:Platform=x64` alone still infers the
invalid `win-AnyCPU` RID and fails `NETSDK1083` (dotnet/sdk#49744), so pass the RID:

```pwsh
cargo build --release --manifest-path core/Cargo.toml
dotnet build apps/windows/ScreenBridge/ScreenBridge.csproj -c Release -p:Platform=x64 -p:RuntimeIdentifier=win-x64
```

For a self-contained, runnable folder that needs no pre-installed Windows App SDK
runtime, publish self-contained:

```pwsh
dotnet publish apps/windows/ScreenBridge/ScreenBridge.csproj -c Release -p:Platform=x64 -r win-x64 --self-contained true -p:WindowsAppSDKSelfContained=true
```

## CI

`.github/workflows/ci.yml` runs the Phase 0 gate on `macos-14` and
`windows-latest`: build core, regenerate + verify the header, clippy/fmt, core
tests, both app shells, and both Swift and C# ping tests.

## Packaging / signing (Phase 5; personal-use default)

Signed/notarised `.dmg` (macOS) and `.msi`/MSIX (Windows) are produced in Phase
5. The default signing target is **personal use** (paid certificates skipped;
steps documented). This is deferred — do not build it in earlier phases.
